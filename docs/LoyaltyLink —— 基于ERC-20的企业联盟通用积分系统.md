# LoyaltyLink —— 基于ERC-20的企业联盟通用积分系统

## 需求分析报告

## 一、选题说明

### 1.1 项目背景与痛点分析

#### 1.1.1 传统企业积分的“三输”困境

积分作为商家激励用户的核心工具，已广泛存在于零售、餐饮、航空、银行等行业。然而，现行积分体系普遍存在以下三大痛点，导致**消费者、商家、社会三方皆输**：

| 利益相关方   | 典型困境                                   | 量化表现（参考行业数据）                                     |
| :----------- | :----------------------------------------- | :----------------------------------------------------------- |
| **消费者**   | 积分价值低、兑换门槛高、容易过期、使用受限 | 国内主流信用卡积分实际价值约为0.02‰~0.1‰；每年约30%积分因过期而浪费 |
| **商家**     | 自建积分系统成本高、用户粘性低、结算复杂   | 中小商户自建积分系统年均成本2-5万元；积分核销率不足20%       |
| **社会整体** | 大量“沉睡积分”造成资源浪费                 | 仅国内航空业每年过期积分价值超10亿元                         |

#### 1.1.2 现有解决方案的局限性

| 方案                         | 优点         | 局限性                                 |
| :--------------------------- | :----------- | :------------------------------------- |
| 单品牌积分（如星巴克星星）   | 用户体验简单 | 无法跨品牌使用，用户被锁定             |
| 聚合积分平台（如支付宝积分） | 兑换范围广   | 中心化平台掌握数据，商家受制于平台规则 |
| 积分联盟（传统模式）         | 可跨商家使用 | 结算对账复杂，缺乏透明度，容易产生纠纷 |

#### 1.1.3 区块链技术的破局价值

区块链，特别是ERC-20通证标准，天然适配积分场景：

- **公开透明**：积分总量、发行规则、每笔交易永久上链，任何人都可验证。
- **用户自主**：积分真正属于用户地址，企业无法单方面作废或贬值。
- **跨主体流通**：任何接入联盟的商家都可接受、消费、结算积分，打破孤岛。
- **可编程**：通过智能合约实现自动分账、过期机制、合规限制等。

### 3.3 核心合约设计

solidity

```
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "@openzeppelin/contracts/token/ERC20/ERC20.sol";
import "@openzeppelin/contracts/access/Ownable.sol";

contract LoyaltyLink is ERC20, Ownable {
    // 商家白名单：地址 → 是否为商家
    mapping(address => bool) public merchants;
    
    // 积分过期配置（可选扩展）
    mapping(address => uint256) public lastActivity;
    uint256 public expiryTime = 365 days;
    
    event MerchantAdded(address indexed merchant);
    event MerchantRemoved(address indexed merchant);
    event Reward(address indexed fromMerchant, address indexed toUser, uint256 amount);
    event Spend(address indexed fromMerchant, address indexed fromUser, uint256 amount);
    
    constructor(string memory name, string memory symbol) ERC20(name, symbol) {}
    
    // 只有管理员可添加/移除商家
    function addMerchant(address merchant) external onlyOwner {
        merchants[merchant] = true;
        emit MerchantAdded(merchant);
    }
    
    function removeMerchant(address merchant) external onlyOwner {
        merchants[merchant] = false;
        emit MerchantRemoved(merchant);
    }
    
    // 商家给用户发放积分（铸造）
    function reward(address user, uint256 amount) external {
        require(merchants[msg.sender], "Only merchants can reward");
        _mint(user, amount);
        emit Reward(msg.sender, user, amount);
    }
    
    // 商家消费扣减积分（销毁）
    function spend(address user, uint256 amount) external {
        require(merchants[msg.sender], "Only merchants can spend");
        require(balanceOf(user) >= amount, "Insufficient balance");
        _burn(user, amount);
        emit Spend(msg.sender, user, amount);
    }
    
    // 可选：积分过期自动销毁（需定期触发）
    function checkAndBurnExpired(address user) external {
        require(block.timestamp >= lastActivity[user] + expiryTime);
        uint256 expiredAmount = balanceOf(user);
        if (expiredAmount > 0) {
            _burn(user, expiredAmount);
        }
    }
}
```



### 3.4 数据流与交互流程

#### 积分发放流程（用户消费 → 获得积分）

1. 用户在线下/线上消费，商家确认金额。
2. 商家打开商户端网页，连接MetaMask（商家账户）。
3. 输入用户地址和积分数量，调用`reward(user, amount)`。
4. MetaMask弹出签名确认，交易发送至Ganache。
5. 交易上链，用户地址积分余额增加。
6. 用户端页面刷新，显示新余额。

#### 积分消费流程（用户兑换商品）

1. 用户选择商品，告知商家欲使用积分。
2. 商家在商户端输入用户地址和扣减数量，调用`spend(user, amount)`。
3. 合约校验用户余额是否足够，若足够则销毁用户积分。
4. 商家确认交易完成，用户获得商品。

#### 用户间转账流程

1. 用户A登录用户端，输入用户B的地址和转账数量。
2. 调用ERC-20标准的`transfer(B, amount)`。
3. 交易上链，双方余额更新。
